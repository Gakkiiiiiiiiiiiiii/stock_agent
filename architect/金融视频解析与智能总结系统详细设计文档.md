## 1. 项目背景

当前智能体平台需要支持对财经视频、个股分析视频、行业研究视频、宏观分析视频、盘前盘后复盘、财报电话会等内容进行自动解析与总结。

普通的视频总结方案通常采用：

```text
视频 → ASR转写 → 大模型全文总结
```

这种方案在金融场景中存在明显问题：

1. 容易遗漏价格点位、时间周期、操作条件和证伪条件。
2. 容易混淆事实、观点、预测、传闻和投资建议。
3. 无法稳定提取支撑位、压力位、目标位、止损位等结构化信息。
4. 对股票简称、代码、行业术语和专业指标识别不稳定。
5. 无法将视频中的语音、字幕、PPT、K线图和表格进行统一时间对齐。
6. 长视频直接总结容易出现上下文遗失和注意力衰减。
7. 输出结果缺少证据时间戳，无法回溯原始视频。
8. 缺少跨视频检索、观点聚合、观点变化分析和历史追踪能力。

因此，本项目不应设计为单一的“视频总结功能”，而应设计为一个完整的：

> 金融多模态视频理解与结构化事件抽取系统。

---

## 2. 建设目标

### 2.1 核心目标

系统需要实现以下能力：

1. 视频音频提取。
2. 场景切分和关键帧抽取。
3. ASR语音转写。
4. 说话人识别。
5. OCR字幕、PPT、表格、股票代码和图表文本提取。
6. 音频文本与画面信息的时间轴对齐。
7. 金融实体识别与归一化。
8. 金融事件结构化抽取。
9. 时间、点位、涨跌、指标、操作建议和风险抽取。
10. 事实、观点、预测、传闻、广告内容分类。
11. 分段总结与全局总结。
12. 关键证据绑定与时间戳回溯。
13. 跨视频语义检索。
14. 同一公司、行业或主题的视频观点聚合。
15. 提供可供智能体调用的 Skill 和 MCP 工具。

### 2.2 非目标

第一阶段不重点实现：

1. 完全自动化证券交易执行。
2. 仅凭视频内容直接生成买卖指令。
3. 对所有K线图进行高精度计算机视觉量化识别。
4. 自动判断主讲人观点一定正确。
5. 以行情验证结果覆盖视频中的原始观点。
6. 未经审核直接将视频观点作为正式投研结论。

## 3. 设计原则

### 3.1 先抽取事件，再生成总结

禁止直接将完整ASR文本交给模型生成最终总结。

推荐流程：

```text
原始视频
  ↓
多模态解析
  ↓
语义切片
  ↓
金融事件抽取
  ↓
实体归一化与校验
  ↓
事件合并
  ↓
结构化摘要
  ↓
自然语言总结
```

### 3.2 Tool负责确定性能力，Skill负责编排与推理

- Tool/MCP：负责视频、音频、OCR、ASR、数据库、实体检索、行情查询等确定性操作。
- Skill：负责判断视频类型、选择领域词典、规划处理流程、调用工具、执行语义抽取和总结。
- LLM：负责语义理解、因果抽取、观点分类和总结。
- 规则引擎：负责数字、时间、单位、代码格式和结果校验。

### 3.3 所有结论都要可回溯

每条关键结论必须包含：

```text
视频时间戳
原始转写文本
OCR证据
画面证据
抽取置信度
```

### 3.4 事实与观点分离

系统必须将内容区分为：

```text
事实
引用数据
主讲人观点
主讲人预测
市场共识
传闻
未经证实信息
投资建议
广告或推广
```

### 3.5 保留条件语义

系统不能仅抽取“看多”“看空”，还应保留：

```text
成立条件
触发条件
失效条件
时间范围
价格范围
仓位建议
风险因素
```

## 5. 核心模块划分

## 5.1 视频接入模块

负责：

1. 视频文件上传。
2. 视频URL下载。
3. 视频元信息获取。
4. 文件哈希计算。
5. 重复视频检测。
6. 任务创建。

输入：

  "source_type": "upload",
  "source_uri": "/data/video/xxx.mp4",
  "title": "快手技术分析",

输出：

## 5.2 视频预处理模块

推荐使用FFmpeg实现。

### 功能

1. 音频提取。
2. 音频转码。
3. 视频分辨率识别。
4. 视频帧率识别。
5. 场景切分。
6. 固定时间间隔抽帧。
7. 场景变化补充抽帧。
8. 黑屏、静帧和重复帧过滤。

### 推荐参数

```text
音频格式：WAV / FLAC
采样率：16000 Hz
声道：mono
关键帧间隔：5～15秒
场景变化阈值：0.25～0.40
```

### 抽帧策略

不建议仅按固定时间间隔抽帧，应采用：

```text
固定间隔抽帧
+ 场景变化抽帧
+ 字幕变化抽帧
+ PPT翻页检测
+ K线画面变化检测
```

## 5.5 多模态统一时间轴

将ASR、OCR、关键帧和视觉标签统一到同一时间轴。

```json
{
  "timeline_id": "tl_001",
  "start_ms": 720000,
  "end_ms": 740000,
  "speaker": "speaker_1",
  "transcript": "快手在三十八块附近有比较强的支撑",
  "ocr_text": "快手-W 1024.HK 日线",
  "visual_tags": [
    "candlestick_chart",
    "MACD"
  ],
  "source_refs": [
    "asr_001",
    "frame_125"
  ]
}
```

多模态融合规则：

1. ASR中出现简称，OCR中出现代码时优先进行联合归一化。
2. ASR数字置信度低，OCR中存在相同数字时提高置信度。
3. 语音与字幕冲突时保留两者，并标记冲突。
4. 图表中的点位不能直接当成主讲人的观点，除非语音或字幕明确提及。
5. OCR仅作为证据和辅助，不应覆盖原始语音含义。

## 12. 事件合并与去重

### 12.1 合并条件

两个事件满足以下大部分条件时可合并：

1. 同一视频。
2. 同一证券实体。
3. 同一事件类型。
4. 时间距离小于预设阈值。
5. 数值接近。
6. 语义相似度超过阈值。
7. 条件和方向不冲突。

### 12.2 示例

原始事件：

```text
12:10：38块是支撑
18:25：跌到38附近我认为可以买
31:45：38不能破，破了结构就坏了
```

合并结果：

```json
{
  "instrument": "1024.HK",
  "support": 38,
  "action": "BUY_NEAR_SUPPORT",
  "invalidation": "跌破38",
  "evidence_timestamps": [
    "00:12:10",
    "00:18:25",
    "00:31:45"
  ]
}
```

### 12.3 冲突处理

若同一视频中出现：

```text
前半段：38是支撑
后半段：38已经失效
```

不能合并成单一结论，应生成：

```text
观点发生更新
前观点失效
后观点覆盖前观点
```

## 17. Prompt设计原则

### 17.1 事件抽取Prompt核心要求

```text
1. 不遗漏价格、百分比、日期、周期和单位。
2. 不将观点改写为事实。
3. 不将传闻改写为确定信息。
4. 保留所有条件和否定关系。
5. 抽取风险、催化剂和证伪条件。
6. 每条事件必须绑定原始时间戳。
7. 不得补充视频中未出现的信息。
```

### 17.2 条件语句处理

原文：

```text
只有放量站稳40港元，才算确认突破。
```

正确输出：

```json
{
  "event_type": "TRADING_ACTION",
  "action": "CONFIRM_BREAKOUT",
  "condition": "放量站稳40港元",
  "status": "NOT_CONFIRMED"
}
```

错误输出：

```text
该股票已经突破40港元。
```

### 17.3 否定语义

原文：

```text
我现在并不认为这里已经见底。
```

必须输出：

```json
{
  "claim_type": "OPINION",
  "sentiment": "BEARISH",
  "statement": "尚未确认见底",
  "negation": true
}
```

## 24. 安全与合规

### 24.1 投资建议风险

最终输出应明确区分：

```text
视频原始观点
系统抽取结果
外部数据验证
系统生成总结
```

系统不能将视频主讲人的观点改写为平台自己的投资建议。

### 24.2 数据安全

1. 视频原文件加密存储。
2. 数据库敏感字段加密。
3. 下载URL设置有效期。
4. 访问按租户隔离。
5. 日志不记录完整视频内容。
6. 支持用户删除视频与派生数据。
7. 支持模型调用内容脱敏。

### 24.3 版权

1. 保存视频来源信息。
2. 限制全文转写导出。
3. 控制长文本原文展示比例。
4. 默认输出摘要与有限证据片段。
5. 支持对私有视频和公开链接采用不同权限策略。

## 30. 最终推荐方案

本功能不应实现为单一Prompt，也不应仅实现为一个简单Skill。

推荐架构：

```text
金融视频理解系统
├─ 视频基础解析服务
├─ ASR与说话人识别
├─ OCR与视觉解析
├─ 多模态统一时间轴
├─ 金融领域本体与词典
├─ 金融事件抽取引擎
├─ 实体归一化与行情验证
├─ 事件合并与证据追踪
├─ PostgreSQL结构化存储
├─ Qdrant语义索引
├─ Reranker结果重排
└─ financial-video-analysis Skill
```

职责边界：

| 组件 | 职责 |
|---|---|
| Skill | 流程编排、视频类型识别、领域选择、总结策略 |
| MCP/Tool | 视频、音频、OCR、ASR、实体查询、行情查询 |
| Schema | 约束事件和总结输出 |
| Lexicon/Ontology | 金融术语、同义词、上下位关系 |
| PostgreSQL | 保存视频、片段、事件、实体和任务 |
| Qdrant | 保存片段、事件和摘要向量 |
| Reranker | 提升跨视频检索和观点聚合质量 |
| LLM | 语义理解、因果关系、观点分类和总结 |
| Rule Engine | 数字、时间、单位、代码、条件和结果校验 |

最终目标不是“把视频压缩成一段话”，而是：

> 将金融视频转换为可查询、可验证、可追溯、可聚合、可进入长期记忆系统的结构化投研知识。
